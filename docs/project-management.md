# Project Management Applications Research - July 2026

## Executive Summary

The project management software market in 2026 is defined by AI-native
workflows, deeper cross-platform interoperability, and heightened demand for
adaptive governance. Over 87% of mid-to-large enterprises use at least two PM
tools simultaneously, reflecting a shift away from monolithic suites toward
purpose-built, composable stacks. SMB adoption has surged to 64% (up from 41% in
2022), citing improved remote collaboration, real-time resource forecasting, and
automated compliance tracking as primary drivers.

This research covers the top 10 project management applications based on G2
Spring 2026 Grid Report, Capterra PM Category Leaderboard, and Forrester Total
Economic Impact studies.

## Strategic Positioning

### The Competitive Gap

Existing leaders are powerful within their archetypes but force customers to
trade off simplicity, speed, structure, or depth:

- **ClickUp, monday.com, Jira**: Praised for customization and breadth but
  criticized for setup burden, feature overload, performance issues,
  administrative overhead, or pricing complexity
- **Asana, Linear**: Praised for clarity and speed but have purposeful
  functional constraints
- **Notion**: Praised for flexibility but criticized for blank-page friction and
  performance degradation at scale

### Product Thesis: Calm Operational Clarity

The optimal position is a personal task manager that can become a team operating
system without forcing every user into enterprise complexity. This follows
directly from the evidence that customers value all-in-one breadth yet resent
confusing setup, clutter, slow performance, rigid structure, and hidden pricing
constraints.

**Differentiation should NOT be**:

- "More views" (ClickUp already emphasizes broad view count)
- "More AI" (all competitors now offer AI features)
- "More custom fields" (monday.com and Jira already lead here)

**Product Thesis**: One underlying work model, five effortless ways to use it:

1. Personal tasks
2. Projects
3. Repeatable processes
4. Client work
5. Team delivery

The system should allow the same work item to function as a personal task,
project task, request, issue, client deliverable, or recurring operational
checklist through templates and context — rather than separate products with
disconnected records.

## Product Principles

### Non-Negotiable Principles

**1. Fast by Default**

- Linear's emphasis on fast, keyboard-first, local-first behavior is the
  meaningful benchmark
- Large workspaces must remain responsive
- Mobile must support real work rather than simple check-ins
- Sub-100ms interactions for core operations

**2. Simple First, Powerful Later**

- Start users with a personal inbox, a daily plan, and a project
- Unlock dependencies, custom fields, workflows, dashboards, and automations
  contextually
- This directly addresses the configuration and learning-curve objections
  attached to ClickUp, monday.com, Jira, and Notion
- Progressive disclosure: show complexity only when user needs it

**3. Structured, Not Rigid**

- Use a clear core hierarchy: Workspace → Project → Section/Phase → Task →
  Subtask
- Allow relationships and multiple views without requiring users to build a
  database schema
- Bridge Asana's clarity with Notion's flexibility
- Opinionated defaults with escape hatches for customization

**4. Collaboration in Context**

- Comments, decisions, files, approvals, and status should live with the
  relevant work item
- Avoid the fragmentation that causes teams to operate across task tools, chat,
  docs, and email
- monday.com strength: closeness-to-work
- Notion strength: documentation-plus-collaboration

**5. Transparent Governance**

- Avoid arbitrary seat blocks, surprise guest billing, opaque action limits
- Avoid essential capabilities hidden behind expensive upgrades
- Each is named as a customer frustration in the research
- Clear, predictable pricing with no hidden costs

## Market Archetypes (Brightfield Research Framework)

The PM market is fragmented across four archetypes:

1. **Flexible Work-Management Suites** - Handle tasks, projects, timelines, and
   reporting tuned for cross-functional teams with fast adoption. (Asana)
2. **Developer-Grade Trackers** - Have issues, sprints, and engineering
   workflows with deep dev integrations. Strong for software teams but heavy for
   others. (Jira, Linear)
3. **All-in-One Configurable Platforms** - Provide highly configurable building
   blocks spanning many use cases. Powerful but can over-complicate. (ClickUp,
   Monday.com)
4. **Docs-First Hybrids** - Combine documents, wikis, and lightweight tasks in
   one. Strong for knowledge work but thin on structured delivery and reporting.
   (Notion)

### Archetype Analysis Table

| Archetype                | Representative      | Strength                                                                           | Structural Weakness                                                                             | Product Lesson                                                                                                                 |
| ------------------------ | ------------------- | ---------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| Flexible work management | Asana               | Clear hierarchy, broad cross-functional fit, fast adoption                         | Constraints on assignment, reporting, financial controls, and some advanced features            | Make accountability and structure obvious, but avoid locking users into one rigid model                                        |
| Configurable all-in-one  | ClickUp, monday.com | Broad views, automations, dashboards, custom workflows, strong consolidation value | Complexity, setup work, feature discovery problems, potentially poor performance at scale       | Build a capable underlying data model, then reveal complexity only when a user needs it                                        |
| Developer-grade trackers | Jira, Linear        | Agile delivery, technical integrations, traceability, workflow rigor               | Either too heavy for ordinary teams or too specialized for broader operations                   | Treat software delivery as an optional operating mode, not the app's core identity                                             |
| Docs-first hybrids       | Notion              | Documentation, flexible databases, asynchronous collaboration, templates           | Weak structured delivery, resource planning, time tracking, and high-scale database performance | Embed contextual documentation and knowledge, but do not make users construct their project-management system from blank pages |

## Personal vs Business User Markets

The research lacks a systematic comparison of these two distinct markets. For an
industry-agnostic tool serving both personal and business users:

### Personal User Requirements

- **Capture speed**: Quick task entry from anywhere
- **Reminders**: Reliable, contextual notifications
- **Recurring tasks**: Easy setup for repeatable personal workflows
- **Offline reliability**: Works without internet connection
- **Calendar behavior**: Seamless integration with personal calendars
- **Privacy**: Data security and privacy controls
- **Low-friction onboarding**: Start using immediately without configuration
- **Mobile quality**: Full-featured mobile experience

### Business User Requirements

- **Roles and permissions**: Granular access control
- **Audit history**: Complete change tracking
- **External collaboration**: Guest access and client portals
- **Data export**: Portable data formats
- **SSO/SCIM**: Enterprise identity management
- **Security**: Compliance certifications, data encryption
- **Reporting**: Dashboards, portfolio views, analytics
- **Integrations**: Deep integration with business systems
- **Billing**: Flexible pricing, invoicing, admin controls
- **Administration**: Centralized management and governance

### Gap Analysis

Current competitors:

- **Trello, Notion**: Strong for personal, weak for business governance
- **Jira, Wrike, Smartsheet**: Strong for business, overkill for personal
- **Asana, Monday.com, ClickUp**: Attempt both but have complexity tradeoffs

**Opportunity**: A system that serves personal users without business
complexity, while scaling to business needs without forcing personal users into
enterprise features.

## Pain-Point Extraction

The research identifies recurring negative signals that can become product
opportunities:

### 1. Feature Overload and Setup Debt

- **Affected**: ClickUp, monday.com, Jira, Notion
- **Issue**: Significant upfront configuration or learning required
- **Opportunity**: Progressive disclosure, smart defaults, template-driven
  onboarding

### 2. Performance and Mobile Gaps

- **Affected**: ClickUp (scale-performance), Notion (large database slowdown),
  monday.com/Notion (mobile weakness)
- **Issue**: Lag, slow loading, limited mobile functionality
- **Opportunity**: Local-first architecture, mobile-first design, performance
  budgets

### 3. Rigid vs Blank-Slate Workflows

- **Affected**: Asana (rigid structure), Notion (intimidating blank canvas)
- **Issue**: Either too constrained or too open
- **Opportunity**: Opinionated templates with customization escape hatches

### 4. Business-Operations Gaps

- **Affected**: Asana, Notion
- **Issue**: Weak in budgeting, invoicing, time tracking, resource management,
  portfolio reporting
- **Opportunity**: Modular business operations features that don't clutter core
  experience

### 5. Pricing Mistrust

- **Affected**: All major tools
- **Issue**: Seat minimums, automation limits, expensive upgrades, guest
  billing, add-on dependency
- **Opportunity**: Transparent pricing, no hidden costs, fair guest access,
  clear upgrade paths

These patterns are more strategically valuable than review scores because they
identify where users may be willing to switch.

## MVP Recommendations

The MVP should be purposefully narrower than the broad suites, but deeper than a
basic to-do list. The goal is to validate whether people value a calm, unified
system before building enterprise surface area.

### Priority Framework

| Priority | Capability                                                                                                                | Why it belongs in MVP                                                                                                                 |
| -------- | ------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| P0       | Personal inbox, quick capture, today/upcoming, recurring tasks, reminders                                                 | Establishes daily personal value and prevents the product from being business-only                                                    |
| P0       | Workspaces, projects, sections, tasks, subtasks, checklists                                                               | Creates a comprehensible hierarchy, reflecting the usability advantages attributed to Asana                                           |
| P0       | Assignees, multi-assignee or accountable-plus-contributors model, due/start dates, priority, status, labels               | Solves fundamental ownership and planning needs while addressing the stated Asana limitation around multiple assignees                |
| P0       | List, board, calendar, and personal "My Work" views                                                                       | Covers the essential working modes without chasing the very high view counts marketed by ClickUp and monday.com                       |
| P0       | Comments, @mentions, attachments, activity history, notifications with strong controls                                    | Supports collaboration while deliberately avoiding the notification overload cited for Asana                                          |
| P0       | Templates and basic task/project duplication                                                                              | Reduces blank-page friction highlighted in the Notion analysis                                                                        |
| P0       | Global search, command palette, keyboard shortcuts, optimistic updates                                                    | Supports the fast interaction standard associated with Linear                                                                         |
| P1       | Dependencies, milestones, timeline, lightweight workload signals                                                          | Adds credible project planning without prematurely replicating enterprise portfolio tools                                             |
| P1       | Forms and simple trigger-action automations                                                                               | Supports intake and repeatable processes, reflecting the value users attribute to monday.com and Asana workflow automation            |
| P1       | Embedded project briefs, meeting notes, decisions, and task-linked documents                                              | Captures Notion's documentation advantage without building a full block-editor platform first                                         |
| P2       | Dashboards, portfolio rollups, goals, time tracking, budgets, client portal                                               | Essential for business expansion, but not necessary to validate core execution behavior                                               |
| P3       | Advanced resource planning, invoicing, complex workflow validators, agile analytics, whiteboards, full database relations | Valuable specialist capabilities, but these are expensive to build and can recreate the complexity users criticize in existing suites |

### MVP Validation Strategy

1. **Personal Value First**: Validate that individuals find daily value before
   team features
2. **Team Transition**: Test how easily personal users can invite collaborators
3. **Progressive Complexity**: Measure at what point users feel the need for
   advanced features
4. **Performance Baseline**: Establish performance metrics for core operations
5. **Mobile Parity**: Ensure mobile experience is not a second-class citizen

## Workflow Analysis by Tool

This section details the workflow and automation capabilities of each analyzed
tool, based on 2026 research.

### 1. ClickUp Workflows

**Automation Architecture**

- Trigger → Condition → Action model
- 100+ pre-built automation templates
- Automation scope: Space, Folder, or List level
- AI-powered automation generation via ClickUp Brain
- Natural language automation description

**Automation Limits by Plan**

- Free: 100 runs/month
- Unlimited: 1,000 runs/month
- Business: 10,000 runs/month
- Business Plus: 25,000 runs/month
- Enterprise: Unlimited

**Key Workflow Patterns**

- Status change → assignee change (handoff automation)
- Due date approaching → automatic reminder/priority change
- New task created → template subtask generation
- Custom field value → task movement/routing
- Recurring task generation with auto-rollover

**Setup Best Practices**

- Designate an Automation Owner account (service account) to prevent permission
  failures
- Enable automation error notifications (off by default in ClickUp 4.0)
- Use descriptive automation names including scope
- Test automations immediately after creation
- Document automations in a team register

**Common Pitfalls**

- Hitting automation caps silently (no dashboard alert by default)
- Personal account authentication breaking Google Drive integrations
- Scope creep (Space vs List level confusion)
- Super Agents generating multiple automation runs per trigger

### 2. Asana Workflows

**Rules Architecture**

- Trigger + Action model (conditions optional)
- Rules gallery with pre-built suggestions
- Custom rule builder with up to 20 triggers per rule
- AI Studio for natural language rule creation
- Rule bundles for reusable automation sets

**Workflow Capabilities**

- Cross-project automation (rules can trigger actions in other projects)
- Integration rules with Slack, Gmail, Jira, Microsoft Teams
- Multiple actions per single rule
- Rule duplication within and across projects
- Automatic AI model selection for AI-powered rules

**Plan Availability**

- Rules available on Starter, Advanced, Enterprise, and Enterprise Plus
- Maximum 50 rules per project regardless of plan
- Rules run on first level of subtasks only (not sub-sub-tasks)

**Setup Process**

1. Navigate to Customize in project top-right
2. Click Rules → + Add
3. Use AI Studio instructions, start from scratch, or select featured rule
4. Configure triggers, actions, and optional conditions
5. Publish rule

**Key Features**

- Section-based rules (lightning bolt icon on sections)
- Rule activity log at project/portfolio level
- Orphaned rule ownership transfer
- Rules can trigger other rules (chain reactions)
- Rule settings: permissions, trigger on subtasks toggle

### 3. Monday.com Workflows

**Automation Architecture**

- Visual "when → then" sentence-based builder
- Board automations (recipes) for simple trigger-action rules
- AI workflows (Pro+ plans) for multi-step, cross-board processes
- AI workflow builder with block-based design
- Sidekick AI assistant for workflow guidance

**Automation Limits by Plan**

- Free: 250 actions/month
- Basic: 250 actions/month
- Standard: 25,000 actions/month
- Pro: 25,000+ actions/month
- Enterprise: Unlimited

**Cross-Board Automations (Pro+)**

- Trigger on any board, action on any other board
- Enables interconnected workspace architecture
- Use cases: deal-to-project handoff, request routing, escalation propagation
- Architectural implication: boards represent workflow stages, not team silos

**AI Workflow Capabilities**

- Plain language workflow description
- Blocks: triggers, conditions, actions, delays
- Call Sidekick action block
- Search the web action block
- Call my agent action block (CRM agents)
- Conditions within conditions

**Date-Based Automations**

- "When date arrives," "X days before date," "X days after date"
- Can set any date, reference other date columns, or build multi-step logic
- Enables precise timeline and dependency management

**Setup Best Practices**

- Map monthly action volume before building
- Use separate date columns for different milestones (planned start, actual
  start, deadline, review date)
- Start with board-level automations, promote to cross-board once validated

### 4. Jira Workflows

**Workflow Architecture**

- Statuses (steps in process) + Transitions (paths between statuses)
- Company-managed: Advanced workflow editor with full configuration
- Team-managed: Simplified workflow editor per work type
- Workflow schemes: Associations between workflows and issue types

**Workflow Customization**

- Multiple workflows per project (different issue types can have different
  workflows)
- Simplified transitions (any-to-any) vs Classic transitions (defined paths)
- Conditions: Restrict who can transition, when to transition
- Validators: Validate field requirements before transition
- Post-functions: Automatic actions after transition

**Status Categories**

- To Do: Work not yet started
- In Progress: Work active or requiring action
- Done: Work complete

**Best Practices**

- Match workflow complexity to issue type needs
- Bugs: Simple workflow (Open → Done)
- Epics: Simplest workflow (Open → Done)
- Stories/Tasks: More complex workflows as needed
- Start with simplified workflows, add defined transitions only when problems
  emerge

**Setup Process (Company-Managed)**

1. Settings → Issues → Workflows
2. Add workflow → Create new
3. Add statuses (allow all transitions initially for flexibility)
4. Apply workflow to project via Project Settings → Workflows
5. Assign workflow to specific issue types

**Key Insights**

- Many teams don't know they can have different workflows per issue type
- Forcing complex workflows on simple issue types creates unnecessary overhead
- The new workflow editor (2024/2025) is more visual but lacks some advanced
  configurations

### 5. Notion Workflows

**Automation Architecture**

- Database automations: Trigger → Action within single database
- Buttons: One-click multi-step actions (manual trigger)
- Recurring templates: Auto-create pages on schedule
- AI autofill: Auto-populate properties from page content
- Formulas 2.0: Computed properties (display only, no side effects)

**Database Automation Triggers**

- Status property changes to specific value
- Date property is reached
- Person property assigned/unassigned
- Checkbox toggled
- Select/multi-select value changes
- Page added to database

**Database Automation Actions**

- Send Slack message to channel/person
- Update property on same page
- Add page to another database
- Send notification to Notion user
- Edit page content (append blocks)

**Key Limitations**

- Automations bound to single database (no cross-database triggers)
- Cannot chain automations (output of one cannot trigger another)
- AI autofill recalculates on database view open, not real-time (slow on large
  databases)
- Recurring templates have no conditional logic (creates whether needed or not)

**Connected Databases**

- Relations: 2-way connections between databases
- Rollups: Pull data across linked databases
- Enables project-task-sprint interconnected systems
- Template: Projects & tasks databases with automatic metrics

**Workflow Setup**

1. Open database options menu → Automations (lightning bolt icon)
2. Select triggers (page added, property edited, any property edited)
3. Choose actions (add page, edit property, send Slack notification)
4. Create automation

**Common Use Cases**

- New task flow: Auto-assign, set status, send Slack notification
- Project kickoff: Auto-create subtasks, notify team
- Progress tracking: Rollup properties calculate velocity automatically

### 6. Linear Workflows

**Cycles (Sprint Planning)**

- Time-boxed iterations (1-8 weeks)
- Auto-generated on schedule once enabled per team
- Cooldown period between cycles (no issues can be assigned)
- Unfinished issues roll forward automatically
- Capacity dial based on last 3 completed cycles' velocity
- Calendar subscription available

**Cycles Configuration**

- Team Settings > Cycles (per team, not workspace-level)
- Set cycle duration (1-8 weeks, 2 weeks default)
- Enable cooldowns after each cycle
- Pick starting day of week (Monday default)
- Set number of future cycles to pre-generate (up to 15)
- Sub-teams inherit parent team's cycle schedule

**Loops (AI-Powered Recurring Workflows)**

- Schedule-based or issue-based triggers
- Natural language instructions for Linear Agent
- Tools for external service communication (GitHub, Notion, Sentry, Slack)
- Workspace-level and team-level loops
- Run history for auditing
- Permissions control access

**Loop Examples**

- When issue enters triage queue, investigate root cause with Code Intelligence
- Every Monday afternoon, review new projects and notify marketing channel
- When issue created in Mobile team, create separate iOS/Android issues

**Git Automations**

- Branch auto-generation from issue IDs
- PR linking and automatic issue closure on merge
- Best-in-class GitHub/GitLab sync

**Best Practices**

- Don't treat mid-cycle priority changes as 1:1 swaps (scope creep)
- Enable estimates if issue sizes vary widely (Fibonacci or T-shirt sizes)
- Use cooldown for retros and backlog grooming without competing against
  delivery

### 7. Smartsheet Workflows

**Automation Architecture**

- Row-based triggers (cell-level changes)
- Formula column triggers (unique capability: automation fires on computed
  conditions)
- Multi-step approval workflows with conditional routing
- Smart Agents and Smart Flows (AI-powered)
- Cross-sheet references and formulas

**Key Differentiator**

- Spreadsheet model enables formula-based triggers
- Build logic in formulas, use automations to respond
- Separation of concerns: formulas for logic, automations for response
- Example: Trigger when calculated Budget Variance column exceeds 15%

**Approval Workflows**

- Multi-step approval chains
- Conditional routing (if Approver A rejects, go to Approver B)
- Approval request emails with inline approve/reject buttons
- Automated reminders to unresponsive approvers
- Full audit trails of approval decisions

**AI Capabilities**

- Smart Agents: Proactively manage tasks, identify risks, recommend next steps
- Smart Flows: Multi-step automations using natural language prompts
- Smart Columns: Analyze sentiment, categorize information, transform data
- Smart Hub: Oversight into AI usage across organization

**Setup Best Practices**

- Define approval chain topology before enabling automations

## AI Assessment

### Current State

The report correctly recognizes AI as a baseline competitive expectation:
ClickUp, Asana, monday.com, Jira, Notion, and Linear all offer some mix of
summaries, drafting, prioritization, triage, workflow creation, risk signals, or
search.

However, the report overemphasizes AI feature parity and does not distinguish
between novelty and durable workflow value.

### AI Principles

**Prioritize AI that reduces coordination labor rather than AI that generates
more content:**

1. **Intake-to-Task Conversion**: Turn notes, emails, or voice capture into
   proposed tasks, owners, due dates, and projects
2. **Status Drafting**: Produce a cited weekly status draft based only on real
   work activity, blockers, and changes
3. **Risk Flagging**: Flag deadline risk from dependencies, stalled work,
   workload, and missing ownership
4. **Explainability**: Explain why an item is overdue or blocked using its audit
   trail
5. **Natural Language Query**: Let users ask natural-language questions about
   work, while always linking the answer back to source tasks, comments,
   documents, and decisions
6. **Human Confirmation**: Require a human confirmation step before AI changes
   assignments, dates, priority, or status

### Anti-Patterns to Avoid

- **Do not make a chat assistant the centerpiece of the interface**: The
  report's main pain signals are overload, complexity, lag, and poor
  discoverability; an ungrounded AI layer would exacerbate all four
- **Do not prioritize content generation over workflow automation**:
  Summarization and drafting are table stakes; coordination reduction is the
  differentiator
- **Do not hide AI behind expensive tiers**: AI should be accessible where it
  provides value, not gated as a premium feature

### AI Implementation Guidance

- **Context-Aware**: AI suggestions must be grounded in actual team behavior and
  work context (like ClickUp, Asana, Linear)
- **Transparent**: Always show the reasoning and source data for AI
  recommendations
- **Opt-Out**: Users should be able to disable AI features without losing core
  functionality
- **Measured**: Track AI adoption and impact; deprecate features that don't
  provide measurable value

## Conclusion

The project management software market in 2026 offers tools for every use case,
but the key is matching the tool to your team's specific needs rather than
chasing feature lists. The four archetypes (flexible suites, developer trackers,
all-in-one platforms, docs hybrids) provide a useful framework for understanding
where each tool fits.

**Key Takeaway**: The best tool is the one your team will actually use. Adoption
friction, learning curve, and fit with existing workflows matter more than
feature count. Start with your team type, size, and primary workflow, then
evaluate tools within that category.

**Recommendation**: For most cross-functional teams, Asana (structured) or
Monday.com (visual) provide the best balance of features, usability, and
adoption. For software teams, Linear (modern/fast) or Jira (enterprise depth)
lead. For knowledge-heavy teams, Notion excels. For budget-conscious teams
needing everything, ClickUp delivers the best value.

## Dependency Validation

The Life OS monorepo includes an automated dependency validation script to
ensure dependency hygiene and prevent common anti-patterns.

### Running the Validation

```bash
# Run dependency validation locally
pnpm validate-deps
```

### Validation Checks

The script performs the following checks:

1. **Workspace Protocol Violations** - Ensures internal dependencies use
   `workspace:` protocol
   - Error: Internal dependency "@life-os/contracts" should use "workspace:"
     protocol, but uses "^0.0.1"
   - Fix: Change version to "workspace:*"

2. **Catalog Mode Violations** - Ensures dependencies in catalog use `catalog:`
   protocol when catalogMode is strict
   - Error: Dependency "zod" is in catalog but uses "^4.0.0" instead of
     "catalog:"
   - Fix: Change version to "catalog:"

3. **TypeScript Version Drift** - Detects inconsistent TypeScript versions
   across packages
   - Warning: TypeScript version drift: @life-os/api uses ^5.9.0, but
     @life-os/vitest-config uses catalog:
   - Fix: Align TypeScript versions across all packages

4. **React/React Native Version Alignment** - Detects inconsistent React
   versions
   - Warning: React version drift: @life-os/ui uses ^18.3.0, but @life-os/web
     uses ^18.2.0
   - Fix: Align React versions across all packages

5. **Missing Test Scripts** - Warns when vitest is in devDependencies but no
   test script exists
   - Warning: Package has vitest in dependencies but no "test" script
   - Fix: Add "test": "vitest" to scripts section

6. **Missing Exports Field** - Warns when packages are missing exports field
   - Warning: Package is missing "exports" field in package.json
   - Fix: Add exports field to package.json

### Fixing Common Issues

**Catalog Protocol Violations**

```json
// Before
"dependencies": {
  "zod": "^4.0.0"
}

// After
"dependencies": {
  "zod": "catalog:"
}
```

**Workspace Protocol Violations**

```json
// Before
"dependencies": {
  "@life-os/contracts": "^0.0.1"
}

// After
"dependencies": {
  "@life-os/contracts": "workspace:*"
}
```

### CI Integration

The validation script runs automatically on all pull requests via GitHub Actions
(`.github/workflows/validate-deps.yml`). PRs with dependency violations will
fail CI checks.

### Severity Levels

- **Error**: Violations that must be fixed (workspace protocol, catalog
  protocol)
- **Warning**: Recommendations that should be addressed (version drift, missing
  scripts)

The script exits with code 1 if any errors are found, causing CI to fail.
Warnings do not cause CI failure but should be reviewed.
