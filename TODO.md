## Purpose and Operating Rules

This task list converts the Work and Calendar module audit into a
dependency-ordered execution plan. It is designed for SDD, DDD, TDD, BDD, and
deep-module implementation.

- **SDD**: Every task begins with repository-specific analysis and research,
  defines a measurable contract, identifies scope boundaries, and names
  validation commands.
- **DDD**: Use domain language such as Workspace, Project, Task, Calendar,
  Event, Scheduling Link, Aggregate, Command, Membership, and Outbox. Keep
  domain invariants near the domain operation that owns them.
- **TDD**: For code changes, write or update a focused failing test before
  implementation, then implement the smallest change that passes, then run the
  listed targeted validation.
- **BDD**: User-facing tasks include Given/When/Then acceptance criteria in
  Definition of Done.
- **Deep modules**: Prefer small stable public interfaces that hide
  authorization, persistence, retries, serialization, and concurrency complexity
  behind focused modules.
- **Assignment**: `AGENT` means the coding agent can perform the work in the
  repository. `HUMAN` is reserved for an external action that cannot be
  performed through repository tooling. No current task requires `HUMAN`
  assignment.
- **Status values**: `ready`, `blocked`, `in_progress`, `done`.
- **Execution order**: Work by dependency order. Within a parent task, execute
  subtasks in listed order unless initial analysis proves the task needs
  correction.
- **Change discipline**: Do not broaden scope during implementation. If analysis
  discovers a new issue, add a new parent task before implementing it.
- **Repository rules**: Follow `AGENTS.md`, especially strict TypeScript, Zod at
  trust boundaries, workspace-scoped authorization, Drizzle migrations, no
  direct client CRUD, transactional audit/outbox writes, and targeted tests.
