## Dependencies to lock now

The project should now create a committed **technology bill of materials** with exact package ranges, compatibility notes, owners, and upgrade rules. Version numbers should not be copied blindly from narrative documents because critical packages evolve quickly and the authoritative answer must be verified against current package registries, release notes, peer dependencies, Expo compatibility, and the generated lockfile at implementation time. For example, current package metadata shows active Hono/OpenAPI, Drizzle, Vitest, Clerk Expo, and Detox ecosystems, but their version lines must be validated as one compatible set rather than selected independently. [npmjs](https://www.npmjs.com/package/drizzle-orm)

The direct production dependency categories should include at least:

- **Backend:** `hono`, Node adapter, Zod, chosen OpenAPI package, Drizzle, `postgres`, JWT/JWKS verification and signing libraries, structured logging, OpenTelemetry SDK/exporters, Sentry SDK, rate limiting, and configuration validation.
- **Database:** Supabase CLI, Drizzle Kit, migration linting, generated database-type validation, PostgreSQL test harness, and SQL formatting/linting tools.
- **Web:** Next.js, React, Clerk Next.js SDK, API client, TanStack Query, form tooling, UI primitives, accessible dialog/popover/menu tooling, and Sentry Next.js SDK.
- **Mobile:** Expo, React Native, Expo Router, Clerk Expo SDK, secure storage, PowerSync, OP-SQLite adapter, push-notification tooling, deep-link/OAuth support, EAS configuration, and Sentry React Native SDK.
- **Shared packages:** contracts, Zod schemas, domain models, API client, command types, date/time rules, and explicitly separated server-only data packages.

| Proof | Dependencies validated | Pass condition | Fallback |

## Deliverable structure

The final research output should be a dependency-selection package with these artifacts:

1. **Master dependency register:** every direct package, hosted service, GitHub Action, Expo native module, CLI, and runtime integration.
2. **Alternatives matrix:** credible alternatives, scored against the product’s architecture and operational constraints.
3. **Exact version manifest:** pinned versions, peer-dependency compatibility constraints, upgrade path, and approved release channel.
4. **Configuration registry:** variables, secret classification, environment ownership, rotation, and exposure policy.
5. **License and supply-chain inventory:** license, maintainer, repository health, SBOM entry, vulnerability scan, and replacement option.
6. **Proof-of-concept plan:** acceptance criteria, test devices, test identities, expected artifacts, fallback trigger, and owner.
7. **Architecture decision record index:** locked, conditional, deferred, rejected, and superseded decisions.
8. **Dependency update policy:** Renovate/Dependabot rules, security-update SLA, compatibility test requirements, and production rollout rules.
