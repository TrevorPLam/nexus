# 10. Risks, Gaps, and Unresolved Questions

Even with a fully researched technology stack and a validated product vision, every ambitious system carries uncertainty. This section catalogs every known risk, architectural gap, and open question that has surfaced through multiple rounds of stress‑testing, research verification, and design iteration. Each entry includes its severity, the impact if left unresolved, and the specific action required to close it.

The risks are organized into four categories: **Technical Integration Risks** (where two chosen technologies must work together), **Architectural & Performance Risks** (inherent tensions in the system design), **Product & Validation Risks** (gaps between what is being built and what has been proven with users), and **Unresolved Decisions** (choices that have been deferred).

---

## 10.1 Technical Integration Risks

### Risk 1: PowerSync + Clerk JWT Compatibility

**Severity:** 🔴 Critical  
**Impact:** If PowerSync cannot accept a Clerk‑issued JWT to authenticate its sync connection, the entire offline‑first mobile data layer breaks. The local SQLite database would not be populated, rendering the mobile app non‑functional. Resolution may require a token exchange service, adding complexity and latency.

**Context:** The architecture assumes that the same JWT used for Supabase RLS can be passed to PowerSync. PowerSync typically expects a Supabase Auth token or a custom token generated by its own service. Clerk’s JWT format is compatible with Supabase RLS (officially documented), but **PowerSync’s acceptance of that same token has not been verified**.

**Recommended Action:** Build a minimal proof‑of‑concept Expo app with Clerk sign‑in, a PowerSync client connected to a Supabase instance, and a Drizzle‑defined schema. Attempt to sync data with RLS enabled. If it fails, design a token‑exchange endpoint that accepts a Clerk token and returns a short‑lived PowerSync‑compatible token, ensuring the exchange can itself be secured.

### Risk 2: Inngest Hosting Location Conflicts

**Severity:** High  
**Impact:** Long‑running background jobs (calendar sync polling, plan recalculation) may exceed Vercel serverless function timeouts if hosted in the Next.js app. If jobs are split between Next.js and the standalone backend, operational complexity increases, and job orchestration becomes harder to reason about.

**Context:** Inngest functions can be hosted in any HTTP server. The directory scaffold currently suggests placing them in `apps/web/inngest/`, but calendar polling and outbox processing are backend concerns.

**Recommended Action:** Consolidate all Inngest job handlers in `apps/backend/src/inngest/`. The web app may still send events to Inngest, but execution will occur in the long‑running backend. This separates frontend deployment from background processing and avoids serverless timeout constraints.

### Risk 3: Dual‑Write to PowerSync and Supabase Realtime

**Severity:** Medium  
**Impact:** Without explicit scoping, a developer might attempt to add Supabase Realtime subscriptions to the mobile app, which would conflict with PowerSync’s own real‑time sync stream. This could cause duplicate UI updates, inconsistent state, and confusing bugs.

**Context:** PowerSync already provides a continuous sync stream for mobile. Supabase Realtime is only needed for the web dashboard, which lacks a local replica.

**Recommended Action:** Document a clear rule in the architecture: mobile real‑time updates come exclusively from PowerSync. Supabase Realtime (or SSE) is a web‑only concern. Enforce this with code reviews and a lint rule if possible.

## 10.2 Architectural & Performance Risks

### Risk 4: Life Graph Query Complexity at Scale

**Severity:** High  
**Impact:** As the number of linked entities grows, a naive “find all tasks due today, their linked projects, contacts, and notes” query will involve many joins and degrade the responsiveness of the Today screen. The core value proposition—instant, calm planning—would be compromised.

**Context:** The solution (pre‑computed materialized views or projection tables updated by the outbox) is designed but not yet implemented. The risk is that the team builds the simple graph first, gains users, and then faces a painful migration when performance degrades.

**Recommended Action:** Implement the `daily_plan_events` projection table from the start, even for MVP, as a lightweight flat table that the planning engine reads. The outbox pattern ensures it stays in sync. This avoids a reactive performance crisis.

### Risk 5: Capacity Engine Rigidity

**Severity:** High  
**Impact:** If the planning engine produces a highly optimized but brittle daily plan that the user cannot follow (due to interruptions, misestimated durations, or shifting energy), the user may feel more stressed, not less. The app could be perceived as an unrealistic taskmaster rather than a helpful partner.

**Context:** The flexibility layer (multi‑plan profiles: Focused, Balanced, Stretch; graceful replanning; commitment levels) is designed but must be built into the engine’s core algorithm, not added later as a patch.

**Recommended Action:** Build the engine with the three plan profiles and the replanning logic from day one, even if the UI only exposes the Balanced plan initially. The engine’s pure‑function I/O contract already supports profile selection; do not defer this complexity.

### Risk 6: Offline Conflict Resolution for Linked Entities

**Severity:** Medium  
**Impact:** When a user completes a task offline that is linked to a project, a goal, and a contact, the cascading updates (project progress, goal check‑in, contact activity log) cannot be applied locally in full. If sync simply overwrites the server state, these cascading effects may be lost or applied inconsistently.

**Context:** The command‑based sync model (complete task command → server replays within a transaction) is designed to solve this. However, the mobile client will optimistically update local projections; mismatches between the optimistic local state and the authoritative server state could cause confusion.

**Recommended Action:** Ensure the command handlers on the server always return the full updated entity graph after processing, and that the mobile client reconciles its local state with the server result. Design a conflict resolution UI (rare but necessary) for cases where a command is rejected (e.g., task already completed on another device).

### Risk 7: Tamagui + Reanimated + PowerSync Performance Overhead

**Severity:** Low  
**Impact:** The Infinite Timeline requires smooth gesture handling (swipe, pinch‑zoom) while a continuous sync stream updates the local database. On lower‑end devices, the combination of Tamagui’s runtime, Reanimated’s UI thread animations, and PowerSync’s data processing could cause jank.

**Context:** This is speculative; both Tamagui and Reanimated are highly optimized. However, it has not been profiled with a realistic dataset.

**Recommended Action:** Profile early in development with a simulated heavy dataset (hundreds of tasks/events per day). Use PowerSync’s incremental loading and lazy queries to avoid loading unnecessary data. If performance issues arise, explore Tamagui’s static extraction or reduce animation complexity.

### Risk 8: Privacy Vault Key Recovery

**Severity:** Medium  
**Impact:** If a user’s encryption key for the Vault Zone is lost (e.g., new device, forgotten master password), all client‑side encrypted data becomes irrecoverable. This could lead to catastrophic data loss and erode trust.

**Context:** The dual‑zone model requires client‑side encryption, meaning the server cannot recover the data. A key recovery mechanism (password‑derived key, recovery phrase, or social recovery) is a product requirement before the Vault Zone is populated.

**Recommended Action:** Design the key management strategy in detail before implementing the Vault. Evaluate password‑derived keys (with Argon2) versus a master password separate from the Clerk account password, combined with a recovery phrase stored securely by the user. This is a UX and security research task.

## 10.3 Product & Validation Risks

### Risk 9: Validation Scope vs. Super‑App Ambition

**Severity:** High  
**Impact:** The validation plan tests the daily planning loop with a concierge pilot, but it does not test whether users will accept cross‑domain integration (finances, health, relationships mixed with tasks). The super‑app hypothesis could pass the MVP validation but fail later when domains are added, leading to costly rework or a product that never achieves its vision.

**Context:** The concierge pilot can inject simulated cross‑domain nudges (fake financial reminders, relationship prompts) to gauge user reaction. This was recommended but is not yet part of the formal validation plan.

**Recommended Action:** Update the concierge pilot protocol to include a small number of non‑task items (bill reminders, health prompts, relationship follow‑ups) and explicitly ask participants whether these felt helpful or intrusive. This early signal will validate or challenge the unification hypothesis before any code for those domains is written.

### Risk 10: Onboarding Complexity and Empty State

**Severity:** Medium  
**Impact:** If a user connects their calendar but has not yet added tasks, the Today screen could appear empty, undermining the value proposition at the critical first‑impression moment. Users may abandon the app before experiencing the planning loop.

**Context:** The MVP requirement states the product must “work even if the user only connects a calendar and enters a rough task list.” The empty state must demonstrate value immediately.

**Recommended Action:** Design the empty state as part of the onboarding wireframes. The initial Today screen with only a connected calendar should show a clear timeline of events plus a gentle prompt (e.g., “What are three things you’d like to accomplish today?”). The weekly review and shutdown flows must also have supportive empty states.

### Risk 11: Weekly Review Adoption

**Severity:** Medium  
**Impact:** The weekly review is essential for system health, but historically, users rarely perform regular reviews in any task management tool. If adoption is low, the system will accumulate stale tasks and projects, degrading the quality of the daily plan over time.

**Context:** The MVP includes a weekly review flow, but it must be exceptionally lightweight and automated to succeed.

**Recommended Action:** Design the weekly review as a guided, low‑friction experience: pre‑populate the agenda (stale tasks, projects without next actions, upcoming calendar), use a progress indicator, and keep the total time under 10 minutes. Consider a “week preview” as an even lighter alternative for users who skip the full review.

## 10.4 Operational & Process Risks

### Risk 12: Meilisearch Operational Overhead

**Severity:** Low  
**Impact:** When the time comes to deploy Meilisearch, it will add a new stateful service to operate, monitor, back up, and scale. If the team is small and focused on product development, this could strain operational capacity.

**Context:** The architecture defers Meilisearch in favor of PostgreSQL FTS for MVP. This is a conscious deferral, not a gap.

**Recommended Action:** Continue with PostgreSQL FTS for MVP. Before Meilisearch is introduced, evaluate whether Meilisearch Cloud (managed) is cost‑effective versus self‑hosting. Document the operational runbook for self‑hosting if that path is chosen.

### Risk 13: Missing Test Infrastructure

**Severity:** Medium  
**Impact:** Without proper test infrastructure from the start, test coverage will lag, and regressions will accumulate. The directory scaffold currently lacks test directories and CI job definitions.

**Context:** The scaffold includes a high‑level CI workflow file but no concrete test configuration.

**Recommended Action:** Add `__tests__/` directories to every package and app, include `jest.config.ts` files extending the base config from `packages/config`, and implement the CI workflow with concrete steps: lint, type‑check, unit tests, and (eventually) E2E tests. Seed the test suite with at least the calendar engine’s edge‑case tests.

### Risk 14: Environment Variable Mismanagement

**Severity:** Low‑Medium  
**Impact:** A missing or misconfigured environment variable can cause hard‑to‑debug runtime errors. With multiple apps and dozens of variables (Supabase, Clerk, PowerSync, Inngest, Sentry, PostHog, Stripe), manual management is error‑prone.

**Context:** The plan calls for Zod‑validated env schemas per app, but these schemas have not been written.

**Recommended Action:** Create `env.ts` files in `apps/web`, `apps/mobile`, and `apps/backend` using `@t3-oss/env-core` with Zod validation. Define shared variable names in `packages/config/env`. Fail fast on startup if required variables are missing. Include these schemas in the repository scaffold.

### Risk 15: Developer Onboarding Friction

**Severity:** Low  
**Impact:** A complex monorepo with many services can slow down new contributors (including future team members) and lead to “works on my machine” issues.

**Context:** The bootstrap script and optional Docker Compose setup address this, but they are not yet built.

**Recommended Action:** Build and test the bootstrap script early. Ensure the Docker Compose stack starts reliably on macOS and Linux. Include a `.devcontainer.json` for VS Code users. Test the onboarding experience with someone who has not seen the repo before.

## 10.5 Unresolved Decisions

These decisions have been intentionally deferred. Each has a defined resolution timeline and does not block MVP development.

| Decision | Options | When to Resolve | Impact of Delay |
|----------|---------|-----------------|-----------------|
| API protocol | tRPC vs. Zod‑validated REST with typed fetch | Before implementing more than a handful of backend routes | Low. The API contract is isolated in `packages/api-client`. Either choice integrates cleanly. |
| Web real‑time mechanism | Supabase Realtime vs. SSE | During web dashboard development (post‑mobile MVP) | Low. The web app initially can use polling or server‑rendered data. |
| Vault key management | Password‑derived keys, master password, recovery phrase | Before any Vault Zone tables are populated | Medium. The Vault Zone is post‑MVP, but the design should be ready before implementation. |
| Workspace mapping (Clerk orgs vs. app workspaces) | Separate app workspaces for V1; map to Clerk orgs later for collaboration | No action needed for V1; decision point when collaboration features are scoped | Low. V1 is single‑user personal workspace. |
